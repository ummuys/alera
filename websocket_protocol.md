# WebSocket protocol

Документ описывает структуры сообщений между frontend и backend на основе текущих файлов `app.js`, `canvas.js`, `events.js`, `ui.js`, `websocket.js`.

Главный принцип протокола: frontend отправляет только намерения пользователя, а backend является источником истины для `clientId`, `nickname`, `roomId`, состава пользователей, истории комнаты, чата, очистки и финальных событий рисования.

## requests

Сообщения, которые frontend отправляет на сервер.

### 1. `join`

```json
{
  type: 'join',
  payload: {
    nickname: this.requestedNickname,
    roomId: this.requestedRoomId,
    color: this.requestedCursorColor,
  },
}
```

#### Поля

| Поле             |     Тип | Обязательное | Описание                                                                                                                  |
| -------------------- | ---------: | -----------------------: | --------------------------------------------------------------------------------------------------------------------------------- |
| `type`             | `string` |                     да | Тип сообщения. Для подключения используется `join`.                                       |
| `payload.nickname` | `string` |                     да | Nickname, который frontend предлагает серверу. Это не доверенная идентичность. |
| `payload.roomId`   | `string` |                     да | Комната, к которой frontend просит подключить пользователя.                            |
| `payload.color`    | `string` |                   нет | Цвет курсора/пользователя, который frontend предлагает серверу. Backend должен проверить и нормализовать значение. |

#### Зачем нужно

Нужно, чтобы frontend сообщил backend-у начальное намерение пользователя: под каким именем он хочет войти, в какую комнату хочет попасть и какой цвет курсора предлагает использовать.

Backend должен проверить и нормализовать эти данные: обрезать пробелы, проверить длину nickname, обработать пустые значения, решить проблему дублей, проверить существование комнаты и права доступа, а также проверить формат цвета. После успешной обработки backend должен вернуть событие `session`.

---

### 2. `draw`

```json
{
  "type": "draw",
  "payload": {
    "x0": 0.12,
    "y0": 0.34,
    "x1": 0.15,
    "y1": 0.36,
    "color": "#111111",
    "size": 5,
    "tool": "pen"
  }
}
```

#### Поля

| Поле          |     Тип | Обязательное | Описание                                                                                                           |
| ----------------- | ---------: | -----------------------: | -------------------------------------------------------------------------------------------------------------------------- |
| `type`          | `string` |                     да | Тип сообщения. Для рисования используется `draw`.                                    |
| `payload.x0`    | `number` |                     да | Начальная X-координата сегмента, нормализованная в диапазоне `0..1`. |
| `payload.y0`    | `number` |                     да | Начальная Y-координата сегмента, нормализованная в диапазоне `0..1`. |
| `payload.x1`    | `number` |                     да | Конечная X-координата сегмента, нормализованная в диапазоне `0..1`.   |
| `payload.y1`    | `number` |                     да | Конечная Y-координата сегмента, нормализованная в диапазоне `0..1`.   |
| `payload.color` | `string` |                     да | Цвет линии. Используется для `pen`.                                                              |
| `payload.size`  | `number` |                     да | Толщина линии или ластика.                                                                           |
| `payload.tool`  | `string` |                     да | Инструмент:`pen` или `eraser`.                                                                            |

#### Зачем нужно

Нужно, чтобы пользователь мог отправить намерение нарисовать один сегмент линии.

Frontend рисует этот сегмент локально сразу через optimistic UI, чтобы пользователь не ощущал задержку сети. Но финальным источником истины всё равно должен быть backend: он валидирует координаты, цвет, размер, инструмент, определяет автора по WebSocket-сессии и рассылает событие `draw` другим участникам комнаты.

Frontend не отправляет `clientId`, `nickname` или `roomId` в `draw`, потому что backend уже знает их из WebSocket-сессии.

---

### 3. `chat`

```json
{
  "type": "chat",
  "payload": {
    "text": "Привет!"
  }
}
```

#### Поля

| Поле         |     Тип | Обязательное | Описание                                                              |
| ---------------- | ---------: | -----------------------: | ----------------------------------------------------------------------------- |
| `type`         | `string` |                     да | Тип сообщения. Для чата используется `chat`. |
| `payload.text` | `string` |                     да | Текст сообщения пользователя.                       |

#### Зачем нужно

Нужно, чтобы пользователь отправил намерение написать сообщение в чат комнаты.

Frontend не добавляет сообщение в чат как успешное сразу. Он очищает input только после успешной отправки в WebSocket и ждёт финальное серверное событие `chat`. Backend должен проверить текст, применить лимиты, удалить или отклонить недопустимые значения, определить автора из WebSocket-сессии и разослать итоговое сообщение участникам комнаты.

---

### 4. `clear`

```json
{
  "type": "clear"
}
```

#### Поля

| Поле |     Тип | Обязательное | Описание                                                                                    |
| -------- | ---------: | -----------------------: | --------------------------------------------------------------------------------------------------- |
| `type` | `string` |                     да | Тип сообщения. Для запроса очистки используется `clear`. |

#### Зачем нужно

Нужно, чтобы пользователь отправил намерение очистить доску.

Frontend не очищает canvas сразу после клика. Он только отправляет запрос на сервер и ждёт server-authoritative событие `clear`. Это защищает от рассинхронизации: если backend запретит очистку или соединение оборвётся, локальная доска не должна отличаться от состояния комнаты на сервере.

## response

Сообщения, которые frontend ожидает или умеет обрабатывать от сервера.

### 1. `session`

Рекомендуемая структура:

```json
{
  "type": "session",
  "payload": {
    "clientId": "client-123",
    "nickname": "User-ab12",
    "roomId": "main",
    "color": "#22c55e"
  }
}
```

Также текущий frontend поддерживает совместимость с такими вариантами:

```json
{
  "type": "session",
  "session": {
    "id": "client-123",
    "name": "User-ab12",
    "roomId": "main",
    "color": "#22c55e"
  }
}
```

#### Поля

| Поле             |     Тип | Обязательное | Описание                                                                                                |
| -------------------- | ---------: | -----------------------: | --------------------------------------------------------------------------------------------------------------- |
| `type`             | `string` |                     да | Тип сообщения. Для подтверждения сессии используется `session`. |
| `payload.clientId` | `string` |                     да | Доверенный идентификатор клиента, назначенный backend-ом.            |
| `payload.nickname` | `string` |                     да | Финальный nickname после серверной нормализации.                             |
| `payload.roomId`   | `string` |                     да | Финальная комната, к которой подключён пользователь.               |
| `payload.color`    | `string` |                   нет | Цвет пользователя для отображения в UI.                                          |

#### Зачем нужно

Нужно, чтобы backend подтвердил успешный join/session handshake и выдал frontend-у доверенную сессию.

После получения `session` frontend сохраняет `clientId`, `nickname`, `roomId`, `color`, показывает текущего пользователя в UI и использует `clientId`, чтобы отличать собственные server echo-события от событий других участников.

---

### 2. `presence`

Рекомендуемая структура:

```json
{
  "type": "presence",
  "payload": {
    "users": [
      {
        "id": "client-123",
        "nickname": "User-ab12",
        "color": "#22c55e"
      }
    ]
  }
}
```

Frontend также распознаёт типы `users`, `onlineUsers`, `online_users`, `online-users`.

#### Поля

| Поле                     |     Тип | Обязательное | Описание                                                                          |
| ---------------------------- | ---------: | -----------------------: | ----------------------------------------------------------------------------------------- |
| `type`                     | `string` |                     да | Тип сообщения. Рекомендуемый тип —`presence`.              |
| `payload.users`            |  `array` |                     да | Полный список пользователей онлайн в комнате.      |
| `payload.users[].id`       | `string` |                     да | Идентификатор пользователя, назначенный backend-ом. |
| `payload.users[].nickname` | `string` |                     да | Отображаемое имя пользователя.                                 |
| `payload.users[].color`    | `string` |                   нет | Цвет пользователя в списке online users.                           |

#### Зачем нужно

Нужно, чтобы backend присылал frontend-у полный актуальный список пользователей онлайн.

Frontend не считает online users самостоятельно по событиям `join` и `leave`. Состав пользователей и счётчик online users должны строиться только из server-authoritative `presence`, чтобы UI не ломался при reconnect, пропущенных событиях, дублях или ошибках соединения.

---

```json
}
```

Frontend также поддерживает типы `userJoined` и `join`.

#### Поля

| Поле            |     Тип | Обязательное | Описание                                                                                  |
| ------------------- | ---------: | -----------------------: | ------------------------------------------------------------------------------------------------- |
| `type`            | `string` |                     да | Тип сообщения. Рекомендуемый тип —`user_joined`.                   |
| `sender.id`       | `string` |                     да | Идентификатор пользователя, который вошёл в комнату. |

| `sender.color`    | `string` |                   нет | Цвет пользователя.                                                                |

#### Зачем нужно

Нужно, чтобы показать системное сообщение о входе пользователя в комнату.

Это событие используется только для уведомления в чате. Оно не должно быть источником истины для online users. Для обновления списка пользователей сервер должен отправлять `presence`.

### 6. `draw`

```json
{
  "type": "draw",
  "sender": {
    "id": "client-456",
    "nickname": "Alice",
    "color": "#3b82f6"
  },
  "payload": {
    "x0": 0.12,
    "y0": 0.34,
    "x1": 0.15,
    "y1": 0.36,
    "color": "#111111",
    "size": 5,
    "tool": "pen"
  }
}
```

#### Поля

| Поле            |     Тип | Обязательное | Описание                                                                                         |
| ------------------- | ---------: | -----------------------: | -------------------------------------------------------------------------------------------------------- |
| `type`            | `string` |                     да | Тип сообщения. Для рисования используется `draw`.                  |
| `sender.id`       | `string` |                     да | Доверенный идентификатор пользователя, который рисовал. |
| `sender.nickname` | `string` |                     да | Nickname пользователя, который рисовал.                                        |
| `sender.color`    | `string` |                   нет | Цвет пользователя.                                                                       |
| `payload.x0`      | `number` |                     да | Начальная X-координата сегмента в диапазоне `0..1`.               |
| `payload.y0`      | `number` |                     да | Начальная Y-координата сегмента в диапазоне `0..1`.               |
| `payload.x1`      | `number` |                     да | Конечная X-координата сегмента в диапазоне `0..1`.                 |
| `payload.y1`      | `number` |                     да | Конечная Y-координата сегмента в диапазоне `0..1`.                 |
| `payload.color`   | `string` |                     да | Цвет линии.                                                                                     |
| `payload.size`    | `number` |                     да | Толщина линии или ластика.                                                         |
| `payload.tool`    | `string` |                     да | Инструмент:`pen` или `eraser`.                                                          |

#### Зачем нужно

Нужно, чтобы backend разослал подтверждённое событие рисования участникам комнаты.

Если `sender.id` совпадает с текущим `session.clientId`, frontend пропускает событие, потому что оно уже было нарисовано локально через optimistic UI. Если событие пришло от другого пользователя, frontend передаёт `payload` в `CanvasHandler.drawDrawPayload(payload)`.

Сейчас основной формат `draw` — один сегмент `x0/y0/x1/y1`. Дополнительно frontend уже умеет принимать будущий stroke-формат с `payload.points[]`; в этом случае `drawDrawPayload` вызывает `drawStroke` и разлагает массив точек на последовательность `drawLine`.

### 8. `room_state`

`room_state` — это полная синхронизация состояния доски. Сервер отправляет его после `join`, reconnect или любого сценария, где клиенту нужно восстановить актуальный canvas.

Основной формат для текущей реализации — история событий:

```json
{
  "type": "room_state",
  "payload": {
    "roomId": "main",
    "version": 42,
    "events": [
      {
        "type": "clear"
      },
      {
        "type": "draw",
        "payload": {
          "x0": 0.12,
          "y0": 0.34,
          "x1": 0.15,
          "y1": 0.36,
          "color": "#111111",
          "size": 5,
          "tool": "pen"
        }
      }
    ]
  }
}
```

Frontend также поддерживает тип `roomState`.

Дополнительно frontend готов к будущему snapshot-формату со strokes:

```json
{
  "type": "room_state",
  "payload": {
    "roomId": "main",
    "version": 43,
    "strokes": [
      {
        "points": [
          { "x": 0.12, "y": 0.34 },
          { "x": 0.15, "y": 0.36 },
          { "x": 0.2, "y": 0.4 }
        ],
        "color": "#111111",
        "size": 5,
        "tool": "pen"
      }
    ]
  }
}
```

#### Поля

| Поле                     |     Тип | Обязательное | Описание                                                                                                                    |
| ---------------------------- | ---------: | -----------------------: | ----------------------------------------------------------------------------------------------------------------------------------- |
| `type`                     | `string` |                     да | Тип сообщения. Рекомендуемый тип — `room_state`.                                                     |
| `payload.roomId`           | `string` |                   нет | Комната, состояние которой отправляется клиенту.                                                         |
| `payload.version`          | `number` |                   нет | Версия состояния комнаты. Нужна для reconnect, диагностики и будущей частичной синхронизации. |
| `payload.events`           |  `array` |                   нет | История или снимок событий комнаты, который нужно воспроизвести на canvas. |
| `payload.events[].type`    | `string` |                     да | Тип события внутри состояния комнаты: `draw` или `clear`.                                    |
| `payload.events[].payload` | `object` |                   нет | Данные события. Для `draw` содержит координаты, цвет, размер и инструмент. |
| `payload.strokes`          |  `array` |                   нет | Альтернативный будущий формат: готовые strokes вместо event log.                         |
| `payload.strokes[].points` |  `array` |                   нет | Массив нормализованных точек stroke-формата.                                                        |

#### Зачем нужно

Нужно, чтобы восстановить актуальное состояние доски после первого подключения или reconnect.

При получении `room_state` frontend больше не содержит replay-логику внутри `WebSocketHandler`. Он передаёт весь payload в `CanvasHandler.drawBoard(payload)`. Canvas-слой сам очищает canvas и последовательно воспроизводит события.

## Общие соглашения

### Нормализация типов сообщений

Frontend нормализует `type`: приводит к нижнему регистру и удаляет пробелы, `_` и `-`.

Поэтому следующие типы будут восприняты одинаково:

```text
user_joined
user-joined
user joined
userJoined
```

Рекомендуется всё равно использовать единый backend-формат в `snake_case`: `user_joined`, `user_left`, `room_state`.

### Sender

Рекомендуемый формат автора события:

```json
{
  "sender": {
    "id": "client-123",
    "nickname": "User-ab12",
    "color": "#22c55e"
  }
}
```

Frontend также временно поддерживает legacy-поля:

```json
{
  "clientId": "client-123",
  "nickname": "User-ab12",
  "color": "#22c55e"
}
```

Рекомендуется использовать `sender`, потому что так автор события явно отделён от payload самого действия.

### Draw payload

Единый payload для событий рисования:

```json
{
  "x0": 0.12,
  "y0": 0.34,
  "x1": 0.15,
  "y1": 0.36,
  "color": "#111111",
  "size": 5,
  "tool": "pen"
}
```

Backend должен валидировать:

- `x0`, `y0`, `x1`, `y1` — числа в диапазоне `0..1`;
- `size` — число в допустимом диапазоне;
- `color` — допустимый цвет;
- `tool` — только `pen` или `eraser`.

### Что frontend не должен отправлять как доверенные данные

Frontend не должен отправлять в пользовательских действиях:

- `clientId`;
- `nickname`;
- количество пользователей онлайн;
- membership комнаты;
- признак того, что очистка уже успешно выполнена;
- server-authoritative историю комнаты.

Эти данные должен назначать, проверять и рассылать backend.


## Frontend board rendering methods

Эти методы добавлены в `canvas.js`, чтобы отделить WebSocket-протокол от логики восстановления canvas. `WebSocketHandler` больше не должен вручную знать, как именно проигрывать историю доски.

### `drawLine(x0, y0, x1, y1, color, size, tool)`

Рисует один нормализованный сегмент линии. Это текущий базовый формат realtime-события `draw`. Координаты ожидаются в диапазоне `0..1`, а сам метод переводит их в размеры текущего canvas.

### `drawStroke(points, color, size, tool)`

Рисует stroke из массива точек. Метод проходит по `points[]` и вызывает `drawLine` для каждой пары соседних точек. Это подготовка к будущему формату, где одна линия передаётся как цельный stroke, а не как множество отдельных сегментов.

### `drawDrawPayload(payload)`

Универсальный метод для одного события рисования. Если в payload есть `points[]`, он вызывает `drawStroke`. Если payload содержит `x0/y0/x1/y1`, он вызывает `drawLine`. Благодаря этому `handleDraw` и `drawBoard` могут работать с обоими форматами.

### `drawBoard(board)`

Полностью восстанавливает canvas из server-authoritative состояния комнаты. Метод сначала вызывает `clearCanvas()`, затем:

1. если есть `board.events`, последовательно проигрывает `clear` и `draw` события;
2. если `events` нет, но есть `board.strokes`, рисует каждый stroke через `drawDrawPayload`.

Этот метод предназначен для `room_state`, а не для обычного realtime-рисования. Обычный `draw` по-прежнему обрабатывается как отдельное маленькое событие.
